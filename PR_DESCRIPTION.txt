# Reachy Mini RoboPay Integration

A paid RoboPay action starts a closed-loop object-inspection mission on the
official Hugging Face Reachy Mini model and validates the behavior across
MuJoCo and Webots.

```text
x402 paid request
  -> Fabric Gateway
  -> RoboPay Tunnel
  -> Zenoh: robot/tunnel/action
  -> Reachy Mini bridge
  -> FSM: SCANNING -> TRACKING -> EXPRESSIVE
  -> MuJoCo execution + Webots sim-to-sim validation
  -> correlated robot/tunnel/result
```

## Simulation task

The Reachy Mini has no arms, so the implemented skill is expressive
head-tracking and table inspection. The new `inspect_table` action tracks
`apple`, `croissant`, and `duck` in sequence.

The policy reads live simulator state (`head_xmat` and target position) at
every step and computes control commands dynamically. No recorded trajectory
or predefined animation is replayed.

## Sim-to-sim validation

The same closed-loop policy is evaluated independently in MuJoCo and Webots.
The Webots run launches the real Webots binary with its controller in batch
mode and reads the result produced by the controller.

## RoboPay payment integration

### Payment safety

- Unpaid `POST /action` returns HTTP `402` with x402 payment requirements.
- Malformed JSON returns HTTP `400`.
- Invalid or unpaid requests publish no `ActionEvent` and cannot move the simulator.
- The Tunnel applies action allowlisting, duration caps, and rate limiting.
- The Tunnel waits for the correlated simulator result before allowing settlement.
- Simulator failure or timeout returns an error and avoids settlement.

### Deterministic positive-payment path

`test_e2e_paid_action.py` uses the real compiled Go Tunnel binary, real Zenoh,
the Reachy Mini bridge, MuJoCo execution, and Webots sim-to-sim validation.
Only the local proxy and facilitator are deterministic test doubles in this
offline test.

```text
paid HTTP request
  -> real Tunnel payment verification
  -> Zenoh ActionEvent
  -> Reachy Mini bridge
  -> MuJoCo result
  -> Webots sim-to-sim result
  -> correlated robot/tunnel/result
```

#### Request flow

1. The client sends an unpaid request to the public Fabric action endpoint.
2. Fabric returns HTTP `402` and the Base Sepolia x402 requirements.
3. The payer signs the returned requirements locally.
4. The signed `PAYMENT-SIGNATURE` is sent to the public endpoint.
5. Fabric routes the request through the public Tunnel WebSocket.
6. The Tunnel verifies payment, publishes the authorized `ActionEvent` to
   `robot/tunnel/action`, and waits for the matching simulator result.
7. The Reachy Mini bridge executes the mission and publishes
   `robot/tunnel/result` with the original correlation identity.
8. Only a successful result permits settlement; the client receives HTTP 200.

## Reproduce

From the repository root:

```bash
make build
make test
python3 bridge/reachy_mini/test_e2e_paid_action.py
```

To run the new multi-object mission locally:

```bash
REACHY_INSPECT_TABLE=1 \
python3 bridge/reachy_mini/test_e2e_paid_action.py
```

Expected result:

```text
paid request status=200
task=multi_object_inspection
objects_completed=3/3
execution_status=SUCCESS
overall_sim2sim_robustness_score=1.0
OK
```

Optional ROS2 execution:

```bash
source /opt/ros/humble/setup.bash
source .venv_ros2/bin/activate
make ROBOT=reachy_mini bridge-build
make ROBOT=reachy_mini bridge-run
```

With the ROS2 bridge already running:

```bash
REACHY_BRIDGE_EXTERNAL=1 \
REACHY_INSPECT_TABLE=1 \
python3 bridge/reachy_mini/test_e2e_paid_action.py
```

The live payer private key is never stored in this repository.
